# 全局多 AI 项目仓库管理规则

日期：2026-05-27

本文档把本项目的多 agent 协作实验抽象成全局规则，适用于 Codex、Claude Code、Trae、Kiro、Goose、Cursor 或其他 AI 编程平台。目标不是绑定某个工具，而是让所有项目仓库都具备同一套可迁移、可交接、可验证的工作秩序。

## 1. 总原则

- 仓库是项目事实源，聊天记录不是事实源。
- `main` 或项目指定主线必须始终可运行。
- 每个 AI agent 只负责一个边界清晰的任务包。
- 每次交接必须留下可读文档，说明做了什么、验证了什么、还剩什么风险。
- 功能行为变化必须有测试或 QA 记录；仅文档变化也要说明为什么无需运行应用测试。
- 外部 AI 平台只能在明确授权的范围内接触仓库信息。
- GUI agent 操作前必须确认当前窗口就是目标项目，不能凭记忆或固定项目名继续执行。

## 2. 每个项目仓库的最低文档契约

每个项目至少应包含以下文档。已有成熟文档可以复用，不要求文件名完全一致，但内容必须可找到。

### 2.1 `AGENTS.md`

用途：所有 AI 平台进入仓库后的第一份操作说明。

必须包含：

- 对用户的语言和沟通偏好。
- 项目当前目标和非目标。
- 当前主线任务入口。
- 分支、提交、测试和文档更新规则。
- 允许使用的项目级 skill 或脚本。
- 高风险操作审批要求。
- 多 agent 协作规则。

### 2.2 `docs/project-overview.md`

用途：让新 agent 在 3 分钟内知道项目是什么。

必须包含：

- 项目定位。
- 目标用户或使用场景。
- 当前已完成能力。
- 核心目录结构。
- 本地启动和验证方式。

### 2.3 `docs/current-goal.md`

用途：记录当前阶段最重要的目标，避免不同 agent 各跑各的。

必须包含：

- 当前版本或阶段目标。
- 本轮优先级。
- 已知阻塞。
- 下一步可执行任务。

### 2.4 `docs/roadmap.md`

用途：记录项目长期路线，避免短期任务破坏长期方向。

必须包含：

- 版本节奏。
- 已完成、进行中、候选能力。
- 重要非目标。
- 未来风险或依赖。

### 2.5 `docs/superpowers/plans/` 或 `docs/plans/`

用途：存放可执行计划。

每个计划应包含：

- 背景和目标。
- 任务拆分。
- 目标文件或模块。
- 验收标准。
- 验证命令。

### 2.6 `docs/superpowers/qa/` 或 `docs/qa/`

用途：存放测试、构建、浏览器检查、手工 QA 或未运行原因。

每条 QA 记录应包含：

- 日期。
- 范围。
- 变更摘要。
- 验证命令和结果。
- 未验证项及原因。
- 已知风险。

### 2.7 `docs/handoffs/`

用途：跨 agent、跨平台、跨天交接。

每条交接记录应包含：

- 日期和分支。
- 任务范围。
- 已改文件。
- 验证结果。
- 阻塞点。
- 风险。
- 下一步建议。

### 2.8 `skills/`

用途：沉淀项目内可复用流程。

适合沉淀为项目 skill 的内容：

- 高频项目流程。
- 容易出错的外部服务流程。
- 固定的 review、发布、QA、数据迁移流程。
- 多 agent 调度协议。

## 3. 分支与版本规则

### 3.1 主线规则

- `main` 是稳定基线，默认必须可启动、可测试、可继续开发。
- 不允许多个 AI agent 直接在 `main` 上并行写代码。
- 从其他分支合入主线前，必须说明测试、文档、风险和回滚方式。

### 3.2 功能分支规则

推荐全局命名：

```text
agent/<platform>/<deliverable>
```

示例：

```text
agent/codex/local-skill-scanner
agent/claude/test-audit
agent/trae/ui-polish
agent/kiro/api-contract
```

如果某个项目已经有稳定约定，例如 `codex/<topic>`，可以继续沿用，但必须写进 `AGENTS.md`。

### 3.3 分支职责

- 一个分支只解决一个交付物。
- 分支名表达交付物，不表达个人情绪或临时想法。
- 扫描、分类、数据转换、推荐逻辑、UI、文档 QA 尽量拆到不同 lane。
- 如果多个分支必须修改同一文件，先明确合并顺序和最终负责人。

### 3.4 提交规则

- 提交前检查 `git status --short`。
- 不提交无关文件、缓存、临时输出、大型生成物。
- 不回滚其他 agent 或用户的改动，除非用户明确要求。
- 提交信息应说明行为变化，不只写“update”。

## 4. 多 AI 平台角色模型

全局协作中，AI 不按平台高低区分，而按职责区分。

### 4.1 项目负责人

职责：

- 读取项目文档和当前状态。
- 拆分任务 lane。
- 决定是否派发给其他 agent。
- 控制分支、合并、提交和风险。
- 汇总最终结论。

默认由当前主对话 agent 承担。

### 4.2 执行 agent

职责：

- 完成一个清晰任务包。
- 只修改授权范围内文件。
- 运行指定验证。
- 返回变更、测试、风险和下一步。

### 4.3 审查 agent

职责：

- 只读检查代码、文档或计划。
- 输出风险、证据和建议。
- 不直接修改仓库，除非另有授权。

### 4.4 GUI agent

职责：

- 操作 IDE、浏览器、设计工具或桌面应用。
- 每次输入或批准前确认窗口项目名匹配目标仓库。
- 遇到项目窗口不匹配时立即停止。

### 4.5 外部 agent

定义：需要把仓库内容、路径、分支、diff、日志或文件片段发给第三方模型或外部服务的 agent。

规则：

- 默认不披露仓库内容。
- 披露前必须说明范围并获得用户明确授权。
- 只披露完成该任务所需的最小信息。
- 外部 agent 输出只作为建议，最终集成由项目负责人完成。

## 5. 派工协议

给任何 AI agent 派工时，应使用短任务、明确边界、明确停止条件。

推荐格式：

```text
目标：完成什么交付物。
范围：允许读取或修改哪些文件。
禁止：不能执行哪些命令或改哪些区域。
验证：必须运行哪些命令。
停止：完成到什么状态就停止。
返回：分支、改动文件、验证结果、风险、未提交状态。
```

反例：

- 一次派发多个阶段的大任务。
- 同时要求切分支、实现、测试、提交、继续下一阶段。
- 只说“帮我优化一下”，不说明验收标准。
- 把禁止命令混在普通命令列表里，导致 agent 误读。

## 6. 命令与审批门禁

### 6.1 默认允许的只读命令

项目可以在 `AGENTS.md` 中列出只读 allowlist，例如：

```text
pwd
ls
rg --files
rg <pattern>
git status --short
git branch --show-current
git diff --stat
git log --oneline -5
```

### 6.2 默认高风险命令

以下操作默认需要项目负责人或用户明确批准：

```text
git checkout
git switch
git stash
git reset
git merge
git rebase
git push
git clean
rm
删除数据库、缓存目录或生成资产
修改生产配置或密钥
```

### 6.3 写入规则

- 写代码前先确认当前分支和工作区。
- 写入范围应与任务 lane 一致。
- 不通过“清理工作区”解决未知变更。
- 遇到他人改动，应先理解意图，再决定如何兼容。

## 7. 验证门禁

每个项目应在 `AGENTS.md` 或 `docs/project-overview.md` 记录验证命令。

推荐分层：

- 快速检查：lint、类型检查、单元测试。
- 功能检查：相关模块测试、接口 smoke test。
- 前端检查：构建、浏览器截图、关键流程点击。
- 阶段收口：全量测试、构建、QA 记录。

交接或合并前必须说明：

- 实际运行了哪些命令。
- 结果是什么。
- 哪些没跑。
- 为什么没跑。
- 后续什么时候补跑。

## 8. 文档同步规则

每轮结束前，项目负责人必须判断是否更新以下内容：

- 当前目标是否变化。
- roadmap 是否变化。
- README 是否需要同步。
- 是否新增 QA 记录。
- 是否新增 handoff。
- 是否形成可复用流程，值得沉淀为 skill。
- 是否影响开源发布、PR、部署或外部用户说明。

文档不是形式主义。它的作用是让下一位 AI agent 不需要猜测上下文。

## 9. GUI 工具安全规则

适用于 Trae、Cursor、Kiro、Xcode、浏览器、设计工具和其他桌面工具。

- 每次操作前确认窗口标题、左上角 workspace 名或侧边栏项目名。
- 目标项目不匹配时立即停止。
- 不要写死某个项目名作为通用流程。
- 不在错误项目窗口里粘贴 prompt、批准命令或修改设置。
- GUI agent 执行后，用仓库状态重新核验，不只相信聊天摘要。

## 10. 信息披露规则

向外部 AI 平台发送以下内容前，必须先说明并获得用户授权：

- 仓库路径。
- 分支名。
- `git status`。
- diff。
- commit log。
- 文件片段。
- 配置、环境变量、日志、报错堆栈。
- 产品路线、客户信息、内部文档。

授权说明应包含：

- 要披露给哪个平台。
- 披露哪些内容。
- 用于什么任务。
- 是否允许工具执行命令。

## 11. 项目整理迁移流程

你要逐步整理所有项目时，建议每个仓库按以下顺序推进。

### Step 1：盘点

- 找到仓库根目录。
- 记录主语言、框架、启动命令、测试命令。
- 检查是否已有 README、AGENTS、docs、skills。
- 检查当前分支和未提交改动。

### Step 2：建立最低文档契约

- 新增或补齐 `AGENTS.md`。
- 新增或补齐 `docs/project-overview.md`。
- 新增或补齐 `docs/current-goal.md`。
- 新增或补齐 `docs/roadmap.md`。

### Step 3：建立协作规则

- 写清主线分支。
- 写清 AI 分支命名。
- 写清高风险命令审批。
- 写清外部 AI 披露规则。
- 写清验证命令。

### Step 4：建立 QA 和交接目录

- 创建 `docs/qa/` 或 `docs/superpowers/qa/`。
- 创建 `docs/handoffs/`。
- 为当前状态补一份 baseline QA 或 handoff。

### Step 5：沉淀项目 skill

- 识别项目高频流程。
- 先沉淀 1 到 3 个最常用流程。
- 每个 skill 保持短小、可执行、可验证。

### Step 6：阶段收口

- 运行项目验证命令。
- 记录未跑项和原因。
- 提交文档基线。
- 再开始功能开发或多 agent 并行。

## 12. 全局检查清单

整理一个项目时，至少回答这些问题：

- 这个项目的主线目标是什么？
- 主线分支是否可运行？
- 新 agent 从哪里开始读？
- 当前最重要的计划在哪里？
- 测试和构建命令在哪里？
- QA 记录在哪里？
- 交接记录在哪里？
- 多 agent 分支怎么命名？
- 哪些命令需要审批？
- 外部 AI 可以看到什么，不能看到什么？
- 哪些流程值得沉淀为项目 skill？

## 13. 推荐全局目录模板

```text
.
├── AGENTS.md
├── README.md
├── docs/
│   ├── project-overview.md
│   ├── current-goal.md
│   ├── roadmap.md
│   ├── plans/
│   ├── qa/
│   └── handoffs/
└── skills/
    └── <project-skill>/
        └── SKILL.md
```

已有项目可以渐进迁移，不要求一次性重构目录。第一优先级是让 AI agent 进入仓库后知道目标、边界、验证方式和风险。
